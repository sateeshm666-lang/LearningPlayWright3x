# Source Code vs Bytecode vs Binary Code

## Example File: `world.js`

```javascript
console.log("Hello World!");
```

## Comparison Table

| Aspect | Source Code | Bytecode | Binary Code (Machine Code) |
|---|---|---|---|
| **Definition** | Human-readable instructions written in a programming language | Intermediate representation between source code and machine code | Raw instructions executed directly by the CPU |
| **Example (world.js)** | `console.log("Hello World!");` | `get_property r0, "console"`<br>`push_string "Hello World!"`<br>`call_method r0, "log", 1` | `48 8b 05 f0 3a 00 00`<br>`48 8d 15 a0 10 00 00`<br>`ff 15 90 35 00 00` |
| **Readability** | Readable by humans | Somewhat readable, but cryptic | Not readable (raw bytes/hex) |
| **Execution** | Cannot execute directly; needs interpreter/compiler | Needs a virtual machine (JVM, V8, .NET CLR) | Executes directly on the CPU |
| **Generation** | Written by developers | Generated by a compiler from source code | Generated by assembler or JIT compiler from bytecode |
| **Portability** | Portable — runs anywhere with the right runtime | Portable across platforms (write once, run anywhere) | **Not portable** — tied to a specific CPU architecture (x86, ARM, etc.) |
| **Optimization Level** | Unoptimized (as written by developer) | Partially optimized by the compiler | Fully optimized for specific hardware |

## How `world.js` Travels Through These Stages

### 1. Source Code (`world.js`)
```javascript
console.log("Hello World!");
```
You write this. It's human-readable JavaScript stored as plain text.

### 2. Bytecode (V8 Ignition Interpreter)
When Node.js (built on V8) runs `world.js`, V8's parser first compiles the source into **bytecode** — an intermediate, compact representation that the Ignition interpreter can execute. You don't see this file; it lives in memory at runtime.

Example internal representation (simplified):
```
LdaGlobal "console"        // Load global 'console'
Star r0                    // Store in register r0
LdaConstant "Hello World!" // Load the string constant
Star r1                    // Store in register r1
CallRuntime JSFunctionCall // Call function
```

### 3. Binary / Machine Code (V8 TurboFan JIT)
If the same code runs many times (it's "hot"), V8's TurboFan JIT (Just-In-Time) compiler kicks in and compiles the hot bytecode into **native machine code** optimized for your specific CPU.

This looks like raw hex bytes:
```
0x00A1B230   48 8b 05 f0 3a 00 00    mov rax, [rip+0x3af0]
0x00A1B237   48 8d 15 a0 10 00 00    lea rdx, [rip+0x10a0]
0x00A1B23E   ff 15 90 35 00 00       call [rip+0x3590]
```

## Key Takeaway

```
 Source Code  ──Parser──►  Bytecode  ──JIT──►  Binary Code
 (world.js)               (Ignition)          (TurboFan)
  Written by              Runs in              Runs directly
  humans                  V8 VM                on your CPU
```

- **Source code**: What you write and edit.
- **Bytecode**: What the VM interprets first (fast startup).
- **Binary code**: What the JIT generates for hot code (fast execution).
